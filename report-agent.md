# ReportAgent v4.0 — ReACT 模式群体智能收敛分析

## 用途

从多轮多 Agent 推演的输出中提取洞察，生成群体智能分析报告。
作为 Layer 2 的 Phase 5 收敛 Agent 使用。

**v4.0 升级**：从"纯文本输入→纯文本输出"升级为 ReACT（Reasoning + Acting）模式——
先规划报告大纲，再逐章节生成，每个章节可调用检索工具获取补充信息。

---

## 工作流程

```
ReportAgent (v4.0):
  │
  ├─ Step 1: 规划大纲
  │   ├─ 输入：所有轮次的观点 + 人格卡 + 配置 + deliberation-log.jsonl
  │   ├─ LLM 输出报告大纲（2-5 个章节）
  │   └─ 规则回退：默认 3 章节（核心发现 / 人群行为 / 趋势与建议）
  │
  ├─ Step 2: 逐章节生成（ReACT 循环）
  │   FOR EACH section IN outline:
  │   ├─ Thought: 这个章节需要什么信息？推演记录中有什么？
  │   ├─ Action: 调用工具获取信息（web_search / read / exec）
  │   ├─ Observation: 分析检索结果，整合进章节
  │   ├─ 重复 Thought→Action→Observation（每章节 2-4 次）
  │   └─ Output: 生成章节正文（Markdown）
  │
  ├─ Step 3: 反思审查
  │   ├─ 某些 Agent 的观点是否未被覆盖？
  │   ├─ 每个结论是否有 Agent 发言作为证据？
  │   └─ 如有遗漏 → 追加一轮补充
  │
  └─ Step 4: 合并输出
      ├─ 拼接所有章节 + 统一输出格式（见下方）
      └─ 输出最终报告
```

---

## 检索工具定义

ReportAgent 作为子 Agent 运行，可使用 OpenClaw 原生工具：

| 工具 | 说明 | 典型用途 |
|------|------|---------|
| `web_search` | 网络搜索 | 验证事实、获取行业背景、寻找数据支撑 |
| `read` | 读取文件 | 读取推演日志、项目文档、历史记录 |
| `exec` | 执行命令 | 数据分析、统计计算 |

在 prompt 中用自然语言描述工具需求即可（子 Agent 会自主决定是否调用）。

## Prompt 模板

### Step 1：大纲规划 Prompt

```markdown
你是群体智能报告的规划专家。根据推演结果，设计最优的报告章节结构。

## 推演问题
{original_question}

## 推演概要
- 参与角色：{N}个 × {M}轮
- 问题类型：{problem_type}
- 收敛情况：第 {converged_round} 轮收敛，最终 stability_rate = {value}

## 核心发现预览
{key_findings_from_all_rounds}

## 任务
输出报告大纲的纯 JSON：

{
  "title": "报告标题（简洁有力）",
  "summary": "一句话摘要",
  "sections": [
    {"title": "章节标题", "description": "内容描述", "data_sources": ["需要检索什么"]}
  ]
}

## 规则
1. 章节数量：2-5 个
2. 每个章节聚焦一个核心发现/分析维度
3. 最后一个章节必须是「建议与行动项」
4. 不要把所有内容塞进一个章节
5. 标题要具体，不要泛泛的「分析」「总结」
```

### Step 2：章节生成 Prompt

```markdown
你是群体智能分析引擎（ReportAgent v4.0），正在撰写报告的一个章节。

## 报告信息
- 报告标题：{report_title}
- 当前章节：{section_title}（第 {section_index}/{total} 章）
- 推演问题：{original_question}

## 已完成章节
{previous_sections_content}

## 全部推演记录（deliberation-log.jsonl）
{formatted_log_data}

## 任务
1. 从推演记录中提取与本章主题相关的核心发现
2. 每个结论必须引用具体 Agent 的发言作为证据
3. 找出共识、分歧、盲区
4. 如果推演数据不足以支撑本章，使用 web_search 搜索补充信息
5. 输出章节正文

## 格式要求
- 使用 **粗体** 标记重点（代替子标题）
- 使用引用块 `>` 展示 Agent 原始发言
- 列表组织要点
- 每段不超过 4 行
- 不要重复已完成章节的内容
- 章节正文 300-600 字
```

### Step 3：反思审查 Prompt

```markdown
你是报告质量审查员。检查以下报告是否存在遗漏。

## 报告全文
{full_report}

## 原始推演数据
{all_rounds_data}

## 审查清单
1. 每个 Agent 的核心观点是否都被引用了？
2. 每个结论是否有 Agent 发言作为证据？
3. 是否有重要的分歧被忽略了？
4. 是否有推演中出现的盲区未被提及？
5. 建议是否有数据支撑？

## 输出
- 如有问题：列出需要补充的内容
- 如无问题：输出 "PASS"
```

## 统一输出格式

最终报告必须包含以下结构（所有章节生成完后，用此格式组装）：

### 🔑 核心发现（Top 3）

从所有碰撞中提取 3 条最关键的洞察。每条必须：
- 有具体证据（引用 Agent 发言）
- 不是泛泛的结论
- 能指导行动

### ✅ 共识区

列出所有或大多数人格都认同的结论。
格式：`共识 + 达成该共识的角色列表 + 证据`

### ⚡ 分歧区

列出无法达成共识的领域。
格式：`分歧点 + 各方立场 + 分歧的根源（利益冲突？信息不对称？价值观差异？）`

### 🕳️ 盲区

所有人格都没覆盖的领域 + 被隐藏假设遮蔽的角落。

### 🌀 涌现模式

碰撞中自发形成的新规律/新洞察（非预设结论）。

### 📊 方案排序

如果原始问题是决策类，输出方案排序：
- 排序维度：可行性、风险、收益、时间线
- 每个方案标注推荐度（⭐-⭐⭐⭐⭐⭐）

### 🏆 推荐方案

最推荐的方案 + 理由 + 风险缓解措施。

### ⚠️ 关键警告

推演中发现的、如果被忽视会致命的风险。

### 📋 下一步

具体的、可执行的行动项（不是泛泛的建议）。

## 与 v3.0 的区别

| 维度 | v3.0 | v4.0 |
|------|------|------|
| 输入 | 纯文本拼接 | 结构化 JSONL 日志 + 收敛数据 |
| 生成方式 | 一次性全文 | 分步 ReACT（大纲→逐章节→反思→合并） |
| 工具调用 | 无 | 每章节 2-4 次（web_search/read/exec） |
| 质量保证 | 无 | 反思审查机制 |
| 证据引用 | 可选 | 强制（每个结论必须有 Agent 发言） |
| 适用场景 | 简单推演 | 复杂多轮推演 |
